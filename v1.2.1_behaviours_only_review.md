# v1.2.1 source-only behaviour-pack review

This review covers the files that exist under `v1.2.1_src_only/behaviors` but are absent at the same relative path under `BOB_behaviors`. The comparison found **9** exact-path candidates.

## Audit method

- Enumerated every file in `v1.2.1_src_only/behaviors` and kept only paths for which `BOB_behaviors/<same-relative-path>` does not exist.
- For every candidate, searched the current main behaviour pack (`BOB_behaviors`) for the candidate identifier, item/block/entity identifier, template-pool identifier, and/or structure path.
- For `.mcstructure` files, treated the structure path without `.mcstructure` as the meaningful worldgen reference and did not recommend moving unreferenced binaries without a matching template-pool or structure-definition route.

## Detailed candidate notes

### 1. `recipes/guidebook.json`

- **Source content:** shapeless crafting-table recipe `better_on_bedrock:guidebook` using `minecraft:dirt` and producing `better_on_bedrock:guide_book`.
- **Current references:** `BOB_behaviors/scripts/commands/guidebook.js` uses the same `better_on_bedrock:guidebook` name for the custom command and grants `better_on_bedrock:guide_book`; the item and placeable guide-book entity already exist.
- **Risk:** the recipe identifier collides semantically with the command identifier but recipe identifiers and command names live in different systems. The real risk is gameplay economy: one dirt for the guide book looks intentionally cheap for onboarding or possibly a debug placeholder.
- **Recommendation:** import only if the desired design is that players can craft the guide book directly. If imported, consider reviewing the ingredient before merging.

### 2. `entities/vanilla/shulker.json`

- **Source content:** a vanilla `minecraft:shulker` override at `format_version` `1.21.70`, including shulker-armour filters that stop shulkers targeting fully equipped players.
- **Current references:** current BP already overrides `minecraft:shulker` at `BOB_behaviors/entities/end_mobs/shulker.json`, and armour recipes/items reference the shulker-armour identifiers.
- **Current-path conflict:** this is not absent functionally; it is absent only by exact relative path. The current shulker file has a different path, older format version, variant component groups, and additional custom targeting behaviour such as reacting to open containers.
- **Recommendation:** do not copy the source-only file into `entities/vanilla/`. If the v1.2.1 changes are desired, cherry-pick specific differences into the current `entities/end_mobs/shulker.json` after testing.

### 3. `entities/vanilla/enderman.json`

- **Source content:** global vanilla `minecraft:enderman` override with health, attack, teleport, looked-at, anger/calm, riding, and shulker-armour filters.
- **Current references:** current BP references `minecraft:enderman` in spawn rules, Ender Dragon behaviour, and bounty scripting, but does not currently include an Enderman entity definition.
- **Risk:** importing this file would replace vanilla Enderman behaviour pack-wide. It is potentially valuable if the shulker armour set is meant to protect against Endermen, but it also changes base mob tuning and AI ownership.
- **Recommendation:** treat as a design review item, not a missing-file restore. If imported, validate Enderman spawning, anger state transitions, player stare detection, water/lava hurt behaviour, and bounty counting.

### 4. `items/ingots/stardust/stardust_block.json`

- **Source content:** item definition `better_on_bedrock:stardust_block` with display name, fire resistance, and `minecraft:block_placer` for block `better_on_bedrock:stardust_block`.
- **Current references:** current BP already defines block `better_on_bedrock:stardust_block`; shaped and shapeless recipes also output/consume that identifier.
- **Risk:** because the block identifier already exists, adding a same-identifier item may be redundant or may alter block-item behaviour. This needs in-game validation rather than blind restoration.
- **Recommendation:** investigate only if the stardust block is not appearing/placing correctly as an inventory item. Otherwise leave absent.

### 5. `worldgen/template_pools/well_dungeon/common/vault.json`

- **Source content:** template pool `pog:common/vault` with elements `well_dungeon/vaults/vault` and `well_dungeon/vaults/ominous_vault`, both using `better_on_bedrock:empty` processors, and a self-referential fallback to `pog:common/vault`.
- **Current references:** current BP has both target structures present, but no current template pool or structure definition references `pog:common/vault`, `well_dungeon/vaults/vault`, or `well_dungeon/vaults/ominous_vault`.
- **Risk:** importing this file alone probably changes nothing because no current jigsaw path reaches the pool. The self-fallback is also worth reviewing, as fallback cycles can make missing-piece behaviour harder to reason about.
- **Recommendation:** import only as part of an intentional well-dungeon route update that references this pool from an existing pool/structure definition.

### 6. `structures/test/vaults.mcstructure`

- **Source content:** small binary `.mcstructure` file under a `test/` path.
- **Current references:** no current main BP reference to `test/vaults` or `vaults` was found.
- **Related current assets:** current BP already contains `structures/well_dungeon/vaults/vault.mcstructure` and `structures/well_dungeon/vaults/ominous_vault.mcstructure`.
- **Recommendation:** do not move. It is an unreferenced binary test asset unless a new structure route is explicitly designed.

### 7. `structures/overworld/overworld_battle_tower.mcstructure`

- **Source content:** large binary `.mcstructure` under `structures/overworld/`.
- **Current references:** no current main BP reference to `overworld/overworld_battle_tower` or `overworld_battle_tower` was found.
- **Related current assets:** current BP has `structures/waystones/overworld_battle_tower.mcstructure` and `structures/overworld/battle_tower.mcstructure`, so battle-tower content is already represented through other paths.
- **Recommendation:** do not move without a template-pool, feature, or structure-definition route that intentionally targets `overworld/overworld_battle_tower`.

### 8. `structures/overworld/overworld_goblin_house.mcstructure`

- **Source content:** binary `.mcstructure` under `structures/overworld/`.
- **Current references:** no current main BP reference to `overworld/overworld_goblin_house` or `overworld_goblin_house` was found.
- **Related current assets:** current BP already includes `structures/overworld/goblin_house.mcstructure` and goblin-outpost structures.
- **Recommendation:** do not move without a route. If this is a newer replacement for `overworld/goblin_house`, compare and intentionally replace the routed asset rather than adding an unrouted path.

### 9. `structures/swamp_village/buildings/house_04.mcstructure`

- **Source content:** binary `.mcstructure` under the source-only `swamp_village/buildings/` namespace.
- **Current references:** no current main BP reference to `swamp_village/buildings/house_04` was found. The current swamp village building pool instead references `village/swamp/buildings/house_04`, and that current structure exists.
- **Risk:** blindly moving this source-only binary would not satisfy the existing route because the routed structure path differs.
- **Recommendation:** do not move. If the source-only binary is desired, intentionally replace `structures/village/swamp/buildings/house_04.mcstructure` or update the template-pool route; do not add the unrouted source path.

## Overall recommendation

Only `recipes/guidebook.json`, `entities/vanilla/enderman.json`, `items/ingots/stardust/stardust_block.json`, and `worldgen/template_pools/well_dungeon/common/vault.json` deserve further human/gameplay review. None should be blindly restored. The four `.mcstructure` candidates should remain unmoved unless a matching worldgen route is deliberately added or an existing routed binary is deliberately replaced.
